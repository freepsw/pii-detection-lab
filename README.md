# PII 탐지 랩 (PII Detection Lab) ㅇㅇ

테이블에 저장된 **개인정보(PII)를 탐지하는 방법은 단일 정답이 없습니다.** 이 랩은 정규식만
쓰는 가장 단순한 방식부터 LLM에 전부 맡기는 방식까지 **5단계 아키텍처**를 차례로 직접
실행해 보면서, 각 방식의 **정확도·비용·운영부담**이 어떻게 달라지는지 수치로 비교하도록
설계됐습니다. PII와 머신러닝/LLM을 처음 접하는 분도 따라올 수 있도록, 개념 입문서와
단계별 노트북을 함께 제공합니다. 이 패키지는 더 큰 레퍼런스 아키텍처를 **교육용으로 단순화**한
자기완결 패키지입니다(운영 코드가 아니라, 아키텍처를 이해해 직접 적용하기 위한 학습용).

> **처음이라면** [`docs/learn/00_여기서_시작.md`](docs/learn/00_여기서_시작.md) 부터 여세요 — 학습·실행
> 순서와 노트북 한눈 지도를 한 곳에 모았습니다.

---

## 1. 이 랩이 다루는 문제

통신사의 테이블에는 두 종류의 PII가 섞여 있습니다.

- **정형(구조화) PII** — 주민등록번호·전화번호·카드번호·계좌번호·이메일·IMEI(단말식별번호)·여권번호처럼
  **컬럼 하나가 통째로 한 종류의 개인정보**인 경우. 형식이 일정해서 정규식으로 비교적 쉽게 잡힙니다.
- **비정형(자유텍스트) PII** — 상담메모·민원·리뷰 같은 **문장 속에 이름·주소·연락처가 섞여 있는** 경우.
  형식이 자유로워 정규식만으로는 한계가 큽니다.

이 랩은 **합성(synthetic) 데이터** 6개 테이블(가입자/청구/통화/직원/상담/리뷰)을 사용합니다.
실제 고객 데이터가 아니므로 안전하게 실험할 수 있으며, **정답 라벨(ground truth)이 생성 시점에
함께 만들어져** 정확도를 객관적으로 측정할 수 있습니다.

## 2. 5단계 아키텍처 한눈에 보기

| 단계 | 구성 | 핵심 메시지 | span exact F1 (gpt-oss) | LLM 라우팅율 | 유효비용 (USD/1k docs) |
|---|---|---|---|---|---|
| **S1** | 패턴매칭(정규식) | 빠르고 거의 공짜. 정형 PII는 잘 잡지만 이름·주소는 0% | **0.7316** | — | **$0** |
| **S2** | 패턴 + LLM | LLM이 비정형(이름·주소)을 회수하고 오탐을 제거 | **0.9448** | 100% | **$0.132** |
| **S3** | 패턴 + NER(사전학습) + LLM cascade | NER이 이름·주소를 저비용 탐지, 저신뢰 구간만 LLM | **0.8966** | 10.9% | **$0.0244** |
| **S4** | 패턴 + NER(파인튜닝) + LLM cascade | 도메인 파인튜닝으로 정확도↑·LLM 호출↓(비용↓) | **0.9838** | **0%** | **$0.01** |
| **S5** | LLM only | 가장 단순하나 S2 대비 정확도 우위 없음 · 비용·운영 의존 최대 | **0.9441** | 100% | **$0.132** |

수치는 모두 **span(텍스트 내 위치 단위) exact-match F1, gpt-oss-120b 백엔드, 평가 코퍼스
4,860 docs 기준**입니다. (트랙·백엔드·분모를 항상 함께 읽는 습관이 중요합니다 — 자세한 읽는 법은
[`docs/learn/04_결과_해석과_단계_선택.md`](docs/learn/04_결과_해석과_단계_선택.md) 참조.)

> **두 가지 LLM 백엔드를 선택할 수 있습니다.**
> - **`databricks-gpt-oss-120b`** — Databricks 호스팅 Foundation Model. 별도 배포 없이 즉시 사용.
> - **`qwen3-4b-t4`** — T4 GPU 커스텀 서빙. **원하는 region(예: Korea Central)에 직접 배포**해
>   데이터가 region 밖으로 나가지 않는 구성이 가능(데이터 레지던시 대응에 유리).
>
> qwen 백엔드는 **선택**입니다. `00_config`의 `qwen_endpoint` 위젯을 비우면 gpt-oss 단독으로 전체
> 랩이 동작합니다.

## 6. 핵심 결과물

랩을 끝까지 실행하면 카탈로그·스키마 아래 다음 테이블이 생성됩니다.

- `arch_evo_comparison` — **stage×backend 단일 비교 그리드**(컬럼·span·비용을 한 장에)
- `col_eval` / `span_eval` — 컬럼/스팬 상세 지표
- `span_ground_truth` / `column_ground_truth` — 정답 라벨(검증 재계산의 기준)
- `text_corpus_masked` / `span_mask_audit` — 거버넌스 산출물(원문↔마스킹)
- `verification_report` — 독립 검증 결과(아키텍처 검증 C1a~C7의 9개 + 패키지 vendoring 무결성 C8 = **10/10 PASS**)

> 위 검증은 `90_eval` 노트북이 런타임에 `verification_report` 테이블로 생성합니다(빌드 자기보고가 아니라 정답 기준 독립 재계산).

## 7. 정직한 해석을 위한 한계 명시

합성 데이터는 **생성기와 탐지기가 같은 포맷 가정을 공유**하므로, S1의 정형 PII 지표(정밀도 1.0)는
**상한치**입니다. 실데이터에는 외국인 주민등록번호(성별자리 5–8), 하이픈 없는 주민번호, 신형 여권
형식, 유선전화(02/031…), 다양한 은행 계좌 포맷 등이 존재하며 현재 정규식은 이들을 놓칩니다.
이 랩에서는 *"정형 PII조차 패턴만으로는 충분치 않다"* 는 **보수적 메시지**로 읽는 것이 정확합니다.

또한 S4 파인튜닝은 **평가 코퍼스와 분리된 학습 코퍼스(seed=777, verbatim 중복 0건)로 학습한 뒤
신규(평가) 데이터에서 측정**한 결과입니다(과거 라벨 데이터로 학습 → 신규 데이터에 적용하는 시나리오).
실데이터 적용 시 기대치와 안전한 비용 하한은 [`docs/learn/06_내_데이터에_적용.md`](docs/learn/06_내_데이터에_적용.md)
에서 다룹니다.
