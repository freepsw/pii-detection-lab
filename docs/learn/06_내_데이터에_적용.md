## 1. 정답 라벨(ground truth)이 없다 — 표본 라벨링이 필요

합성 데이터는 PII를 심으면서 **정답 라벨이 공짜로** 만들어졌습니다(offset까지 보장). 실데이터에는
이 정답표가 **없습니다.** 그래서 두 가지가 막힙니다.

- **정확도 측정 불가** — 정답이 없으면 F1·recall을 잴 수 없습니다.
- **파인튜닝 불가** — 학습에 쓸 라벨이 없습니다.

**해법** — 실데이터에서 **대표 표본을 사람이 직접 라벨링**합니다(보통 수백~수천 문서). 라벨링한 표본을
ⓐ 일부는 **평가용**(정확도 측정), ⓑ 일부는 **학습용**(파인튜닝)으로 나눕니다. 라벨링 효율을 위해 S2/S5
LLM 예측을 **초안**으로 띄우고 사람이 교정하는 방식(human-in-the-loop)을 권장합니다.

> **라벨링 공수 감(感) 잡기** — 이 랩 규모는 학습 2,946 / 평가 4,860 문서입니다. 실프로젝트는 보통 평가용
> 수백 건 + 학습용 수백~수천 건이면 시작할 수 있고, LLM 초안+사람 교정 시 문서당 약 1~3분이 듭니다
> (예: 학습 1,000건 ≈ 20~50시간). 정확한 산정이 아니라 **규모 감을 잡기 위한 어림치**입니다.

### 대표 표본을 어떻게 뽑나 — 층화 추출(stratified sampling) 예시

표본이 한쪽 테이블·한쪽 PII 유형에 쏠리면 평가·학습 모두 편향됩니다. **테이블별·PII 밀도별로 고르게**
뽑는 층화 추출을 권장합니다. 아래는 *여러분의 환경에 맞춰 고칠 예시 SQL*입니다(테이블·컬럼명은 일반화한
예시이며, PII 밀도는 정형 PII 정규식이 자유텍스트 컬럼에서 1건이라도 매칭되는지로 근사합니다).

```sql
-- 예시: 테이블 × PII 밀도(고/저) 층마다 같은 수를 뽑아 대표성 확보 (의사 SQL — 환경에 맞게 수정)
WITH labeled AS (
  SELECT
    doc_id,
    table_name,
    consult_note AS text,                                  -- 라벨링 대상 자유텍스트 컬럼(예시명)
    CASE WHEN consult_note RLIKE '[0-9]{6}-[1-4][0-9]{6}'  -- 정형 PII 흔적 = '밀도 높음' 근사
              OR consult_note RLIKE '01[016789]-?[0-9]{3,4}-?[0-9]{4}'
         THEN 'pii_dense' ELSE 'pii_sparse' END AS pii_stratum
  FROM your_catalog.your_schema.your_source_table          -- 여러 소스 테이블이면 UNION ALL로 결합
),
ranked AS (
  SELECT *,
         ROW_NUMBER() OVER (PARTITION BY table_name, pii_stratum ORDER BY rand(42)) AS rn
  FROM labeled
)
SELECT doc_id, table_name, text, pii_stratum
FROM ranked
WHERE rn <= 150        -- 층(table_name × pii_stratum)마다 150건 → 표본을 고르게 분산
ORDER BY table_name, pii_stratum, rn;
```

뽑은 표본을 라벨링한 뒤 학습/평가로 나눕니다(분리 규율은 §2 참조). `rand(42)` 처럼 시드를 고정하면
표본 추출이 재현됩니다.

## 2. 재파인튜닝 — 자체 라벨로 S4를 다시 학습

S4의 강점(라우팅 0%·정확도 최고)은 **우리 도메인 라벨로 NER을 파인튜닝**했기 때문입니다. 실데이터의
이름·주소 표기·문체는 합성 데이터와 다르므로, **반드시 자체 라벨 데이터로 재파인튜닝·재검증**해야 합니다.

- 이 랩의 파이프라인을 그대로 재사용하되, 학습 코퍼스를 **여러분의 라벨 표본**으로 교체합니다 — 구체적으로
  `40_S4_pattern_nerft_llm/S4a_generate_train_corpus`(합성 생성 대신 내 라벨을 `text_corpus_train`/`span_gt_train`에
  적재) → `S4b_ner_finetune_cascade`(재학습) → `90_eval`(재측정) 순으로 실행합니다.
- **학습-평가 분리는 필수**입니다 — 학습에 쓴 문서가 평가에 섞이면(data leakage) 점수가 부풀려집니다.
  이 랩이 seed=777·verbatim 중복 0건으로 분리한 것과 동일한 규율을 지키세요.
- 라벨이 적으면(수백 건) 사전학습 모델 위에 파인튜닝하는 것만으로도 큰 향상을 얻습니다(⑥).

### 라벨 스키마 — 내 라벨을 S4a 파이프라인에 맞추는 법

S4a 파이프라인에 자체 데이터를 넣으려면 **span 라벨 컬럼이 이 랩의 `span_ground_truth` 와 같은 모양**이어야
합니다. 정답표(`data/span_ground_truth.csv`)의 실제 컬럼은 다음과 같습니다.

| 컬럼 | 의미 |
|---|---|
| `doc_id` | 문서 고유 ID(예: `테이블.컬럼.row_key` 형태) — 학습/평가 분리와 추적의 키 |
| `table_name` | 이 PII가 나온 소스 테이블명 |
| `column_name` | 이 PII가 나온 컬럼명(자유텍스트 컬럼) |
| `row_key` | 원본 행 식별자(테이블 PK) |
| `start_char` | PII span 시작 글자 offset(④ 참조, 0-기준) |
| `end_char` | PII span 끝 글자 offset |
| `entity_type` | PII 유형(예: `RRN`·`PHONE`·`PERSON`·`ADDRESS` 등) |
| `pii_value` | 해당 span의 실제 PII 문자열(offset 검증용) |
| `source_layer` | 라벨 출처 레이어(예: `PATTERN`·`NER`·`LLM`) |

컬럼 트랙 라벨(`column_ground_truth`)은 더 단순합니다 — `table_name`, `column_name`, `is_pii`(true/false),
`category`(예: `PERSONAL_INFO`·`NON_PII`) 4개 컬럼입니다. 즉 여러분의 라벨을 위 두 스키마(span/컬럼)에
맞춰 `text_corpus_train`/`span_gt_train` 로 적재하면 S4a 이후 단계가 그대로 동작합니다. **offset(`start_char`/
`end_char`)은 `pii_value` 와 실제 텍스트가 글자 단위로 일치**해야 검증(C1a)을 통과합니다(④).

## 3. 실데이터 기대 정확도와 비용 하한 보장

### 기대 정확도 (보수적으로)

합성 데이터의 S4 0.9838은 상한치입니다. **실데이터에서는 약 0.85~0.92 수준**을 보수적 기대치로
잡으세요(도메인·라벨 품질·표기 다양성에 따라 변동). 처음부터 0.98을 기대하지 말고, 표본 평가로
실측한 뒤 라벨을 늘려가며 개선하는 접근이 안전합니다.

### 비용 하한 보장 (열화 안전성)

정확도는 데이터에 따라 변하지만, **비용에는 구조적 하한·상한이 있습니다.**

- **하한** — S4처럼 cascade 라우팅이 0%에 가까우면 비용은 **약 $0.01/1k docs**(NER 추론 오버헤드)에
  수렴합니다. 실데이터에서 NER이 덜 자신 있어 라우팅이 늘어도, **최악의 경우(라우팅 100%)에도 전수
  LLM인 S2/S5와 같은 $0.132/1k docs를 넘지 않습니다.**
- 즉 cascade를 쓰는 한 **유효비용은 $0.01 ~ $0.132 사이**에 머뭅니다. 파인튜닝이 덜 효과적이어도
  비용이 폭증하지 않는다는 **열화 안전성(graceful degradation)** 이 cascade 구조의 장점입니다.

> 비용은 예시 단가 기반 추정·LLM 처리비 한정입니다. 실제 단가·토큰 수에 따라 절대값은 조정하세요.

## 4. 정규식 변형 — 실데이터 형식에 맞춰 확장

이 랩의 정규식 9종(`_common/span_patterns.py`)은 합성 데이터의 **표준 형식**만 가정합니다. 실데이터에는
다음 변형이 흔하며, 현재 정규식은 이들을 **놓칩니다.** 실데이터 적용 시 패턴을 보강하세요.

| PII | 합성 데이터(현재 정규식) | 실데이터 변형(추가 필요) |
|---|---|---|
| 주민등록번호 | 내국인 성별자리 1–4 (`\d{6}-[1-4]\d{6}`) | **외국인 성별자리 5–8**, **하이픈 없는** 13자리 |
| 여권번호 | `[MSROD]\d{8}` | **신형 여권**(예: `M123A4567` — 영문+숫자 혼합) |
| 전화번호 | 휴대폰 `01[016789]-…` | **유선전화**(02/031/051 등 지역번호) |
| 계좌번호 | `\d{3}-\d{2}-\d{6}` 한 형식 | **은행별 다양한 계좌 포맷**(자릿수·구분자 상이) |

> 정규식은 추가할수록 정형 PII 재현율이 오르지만, 과탐 위험도 같이 커집니다. 변형 추가 후에는
> 표본 평가로 정밀도가 떨어지지 않는지 확인하세요. **정형이라도 패턴만으로는 충분치 않다**는 것이
> 이 랩의 보수적 메시지이며, 비정형은 여전히 NER/LLM(S2~S4)이 필요합니다.
>
> 🔄 **이 변형들이 운영 중 라벨 없이 유입될 때** 자동으로 탐지·교정하는 Day-2 폐곡선은

## 5. 백엔드 선택 (in-region 여부)

- **데이터 레지던시 요구가 없다** → `databricks-gpt-oss-120b`(호스팅 FM). 즉시 사용·고성능.
- **데이터가 region 밖으로 나가면 안 된다** → `qwen3-4b-t4` 를 해당 region(예: Korea Central)에 배포해
  백엔드로 지정(01_환경_설정 ④). **S4는 라우팅 0%라 LLM 의존이 최소**여서 in-region 처리량 제약의
  영향이 가장 작습니다.

특정 법령(국외이전 규제 등) 충족 여부는 법무 검토 사안입니다. 이 랩은 기술적으로 **데이터가 region을
벗어나지 않는 구성이 가능함**을 보일 뿐입니다.

## 6. 적용 체크리스트

실데이터 적용 프로젝트를 시작할 때 아래를 따르세요.

- [ ] **대표 표본 선정** — 테이블·문서 유형을 고르게 포함한 수백~수천 문서
- [ ] **표본 라벨링** — 컬럼 라벨 + 텍스트 span 라벨(offset 포함), LLM 초안+사람 교정 권장
- [ ] **학습/평가 분리** — verbatim 중복 0건 확인(data leakage 방지)
- [ ] **정규식 변형 보강** — §4 표(외국인 RRN·하이픈 없는·신형 여권·유선전화·계좌 포맷)
- [ ] **S4 재파인튜닝** — 자체 학습 표본으로, 분리 평가 표본에서 실측
- [ ] **기대치 보수 설정** — 0.85~0.92 목표로 시작, 라벨 늘리며 개선
- [ ] **비용 추정** — 라우팅율 실측 → 유효비용($0.01~$0.132 범위) 산정
- [ ] **백엔드 결정** — in-region 필요 여부에 따라 gpt-oss vs qwen
- [ ] **거버넌스 검증** — 마스킹 후 잔존 PII 0건([`05_거버넌스_마스킹.md`](05_거버넌스_마스킹.md) §4 검증 SQL) + 독립 검증 통과 확인
- [ ] **재현성 고정** — seed·버전 핀으로 동일 재현 보장
