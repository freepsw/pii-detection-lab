## ① Unity Catalog 카탈로그·스키마·Volume 권한

이 랩은 **하나의 카탈로그 + 하나의 스키마** 안에서 모든 테이블·Volume을 생성합니다.
설정의 **단일 출처(single source of truth)** 는 `00_setup/00_config` 노트북의 **위젯 4개**(위젯 = 노트북 상단의 입력 칸)입니다.
이 위젯값만 바꾸면 전체 랩에 전파됩니다(다른 노트북은 첫 셀에서 `%run ../00_setup/00_config` 로 상속).

| 위젯 | 설명 | 예시 |
|---|---|---|
| `catalog` | 작업할 Unity Catalog 카탈로그 | `main` |
| `schema` | 작업 스키마(없으면 자동 생성 시도) | `pii_lab` |
| `gptoss_endpoint` | Foundation Model 엔드포인트명 | `databricks-gpt-oss-120b` |
| `qwen_endpoint` | (선택) in-region 커스텀 서빙명. **비우면 gpt-oss 단독 실행** | (비움) |

`00_config` 노트북은 마지막 셀에서 다음을 **자동 시도**합니다.

```sql
CREATE CATALOG IF NOT EXISTS <catalog>;     -- 권한 없으면 스킵(기존 카탈로그 사용)
CREATE SCHEMA  IF NOT EXISTS <catalog>.<schema>;
CREATE VOLUME  IF NOT EXISTS <catalog>.<schema>.raw;     -- CSV 업로드용
CREATE VOLUME  IF NOT EXISTS <catalog>.<schema>.models;  -- NER 모델 저장용
```

**필요한 권한**

| 작업 | 필요 권한(예시) |
|---|---|
| 카탈로그 생성 | metastore에 `CREATE CATALOG` (없으면 관리자가 미리 만들고 위젯에 지정) |
| 스키마·테이블 생성 | 카탈로그에 `USE CATALOG` + `CREATE SCHEMA`, 스키마에 `CREATE TABLE` |
| Volume 생성·쓰기 | 스키마에 `CREATE VOLUME`, Volume에 `READ/WRITE VOLUME` |
| 거버넌스(태그·마스크) | 대상 테이블 `OWNER` 또는 `MODIFY` + `APPLY TAG` |

> **권한이 없다면?** 관리자에게 **카탈로그·스키마·Volume 2개(raw·models)** 를 미리 만들어 달라고
> 요청한 뒤, `catalog`/`schema` 위젯에 그 이름을 입력하세요. `00_config`의 `CREATE … IF NOT EXISTS`
> 는 이미 존재하면 조용히 넘어갑니다.

## ② ML 런타임 17.3 클러스터 (CPU, GPU 불필요)

NER 추론·파인튜닝(S3/S4)은 **Databricks Runtime 17.3 LTS for Machine Learning** 위에서 동작합니다.
**GPU는 필요 없습니다** — CPU 노드로 KoELECTRA 계열 NER을 추론·파인튜닝합니다(약 25분 내외).
S1/S2/S5와 거버넌스·평가는 일반 컴퓨트로도 충분하지만, 단일 클러스터로 전 단계를 돌리려면 ML 17.3을
권장합니다.

**클라우드별 권장 노드 타입** (16 vCPU / ~128GB 메모리급, 로컬 SSD 포함)

| 클라우드 | 권장 노드 타입 | 비고 |
|---|---|---|
| **Azure** | `Standard_D16ds_v5` | 16 vCPU, 64GB, 로컬 SSD |
| **AWS** | `m5d.4xlarge` | 16 vCPU, 64GB, NVMe SSD |
| **GCP** | `n2-highmem-16` | 16 vCPU, 128GB |

> **GPU를 굳이 쓰지 않는 이유** — 이 랩의 NER 모델은 작고(파라미터 수억 규모), 데이터도 수천 문서
> 규모라 CPU로 충분히 빠릅니다. GPU는 ④의 **LLM 서빙(qwen3-4b-t4)** 에만 쓰이며, 그조차 별도
> 서빙 엔드포인트로 분리되어 있어 이 잡 클러스터에는 GPU가 필요 없습니다.

단일 노드(single-node) 구성으로 충분하며, 자동 종료(auto-termination)를 짧게(예: 30분) 두면 비용을
아낄 수 있습니다.

## ③ Foundation Model `databricks-gpt-oss-120b` region 가용 확인

S2/S3/S4/S5의 LLM 호출은 기본적으로 Databricks 호스팅 Foundation Model
**`databricks-gpt-oss-120b`** 를 사용합니다(SQL에서는 `ai_query`, Python에서는 serving 호출).

**가용 확인 방법**

1. 워크스페이스 좌측 **Serving** 메뉴에서 `databricks-gpt-oss-120b` 엔드포인트가 보이는지 확인,
   또는
2. 노트북에서 간단히 호출 테스트:
   ```python
   from databricks.sdk import WorkspaceClient
   from databricks.sdk.service.serving import ChatMessage, ChatMessageRole
   w = WorkspaceClient()
   r = w.serving_endpoints.query(
       name="databricks-gpt-oss-120b",
       messages=[ChatMessage(role=ChatMessageRole.USER, content="ping")],
       max_tokens=8)
   print(r)
   ```

**미가용 시 대체법**

- **다른 Foundation Model로 교체** — `00_config`의 `gptoss_endpoint` 위젯에 해당 워크스페이스에서
  사용 가능한 instruct 계열 엔드포인트명을 입력합니다(JSON 출력을 따르는 모델이면 동작).
- **qwen 백엔드만 사용** — ④를 배포하고 `qwen_endpoint` 위젯에 그 이름을 넣은 뒤, gpt-oss 행은
  비워 둡니다. 단, **SQL `ai_query` 배치 경로는 커스텀 엔드포인트를 지원하지 않으므로**(아래 ④ 주의)
  qwen은 Python 호출 경로로 동작합니다.

> 비용·정확도 수치(README의 비교표)는 **gpt-oss-120b 기준**입니다. 다른 모델로 교체하면 수치가
> 달라질 수 있습니다.

## ④ (선택) `qwen3-4b-t4` 커스텀 서빙 배포 — in-region 데모용

> **먼저 — 이 단계는 선택이자 고급(in-region) 구성입니다.** 랩 전체는 `databricks-gpt-oss-120b` 단독만으로 완주할 수 있으며, qwen 배포는 **데이터 레지던시 요건이 있을 때만** 진행하세요. 그 경우 아래 주의사항을 먼저 읽거나 담당 Databricks SA에 문의하시기 바랍니다.

데이터가 **region 밖으로 나가지 않아야 하는** 요구(데이터 레지던시)가 있다면, 작은 LLM을 원하는
region(예: Korea Central)에 직접 배포해 백엔드로 쓸 수 있습니다. 이 랩은 **Qwen3-4B**를 T4 GPU
서빙으로 배포하는 구성을 지원합니다.

- **모델**: `Qwen/Qwen3-4B-Instruct-2507`
- **워크로드 타입**: `GPU_SMALL` (T4), `scale_to_zero` 가능
- **엔드포인트명(예시)**: `qwen3-4b-t4` → `00_config`의 `qwen_endpoint` 위젯에 입력

**배포 핵심 메모(검증됨)**

- 등록(모델 logging)은 **17.3 LTS ML CPU**, 서빙은 **T4(fp16)** 로 분리.
- 콜드 스타트는 최초 약 15~32분, 재기동은 1분 남짓.
- pyfunc 직렬화 함정: `__getstate__` 로 tokenizer/model 객체를 pickle 대상에서 제외, dtype은
  GPU에서 fp16 / CPU에서 bf16 분기, `pip_requirements` 에 torch를 명시하지 않음(서빙 이미지가 제공),
  transformers 버전 핀.

> **중요 — SQL `ai_query` 미지원**: 커스텀 서빙 엔드포인트는 SQL `ai_query` 에서 호출할 수 없습니다
> (`REMOTE_FUNCTION_HTTP_RESULT_UNEXPECTED_ERROR`). 따라서 qwen 백엔드의 컬럼/span 배치는 Python
> 호출 경로(`_common/llm_client.py`)로 처리되며, gpt-oss보다 처리량이 낮습니다(단일 T4 replica 직렬
> 약 0.49 rps). 대량 처리에는 replica 증설 또는 **S4 파인튜닝 NER로 LLM 의존을 0%까지 줄이는** 구성을
> 권장합니다.

> qwen 커스텀 서빙은 이 랩의 **선택(opt-in)** 고급 항목입니다. 기본 경로(gpt-oss 단독)로 S2~S5를
> 완주할 수 있으며, in-region(데이터가 region 밖으로 나가지 않는 구성) 데모가 필요할 때만 배포합니다.
> 위 5단계 요약으로 배포가 가능하며, 더 상세한 절차가 필요하면 담당 Databricks SA에 문의하세요.

## ⑤ (S3/S4 실행 시) HuggingFace 모델 다운로드 허용

S3(사전학습 NER)와 S4(파인튜닝 NER)는 **KoELECTRA 계열 한국어 NER 모델**을 HuggingFace에서
내려받아 사용합니다. 잡 클러스터가 **외부 인터넷(HuggingFace Hub)에 접근**할 수 있어야 합니다.

**폐쇄망(인터넷 차단) 환경 대안**

1. 인터넷이 되는 환경에서 모델을 미리 내려받아 **UC Volume(`<schema>.models`)** 에 업로드합니다.
   ```python
   from huggingface_hub import snapshot_download
   snapshot_download("monologg/koelectra-base-v3-discriminator",
                     local_dir="/Volumes/<catalog>/<schema>/models/koelectra-base")
   ```
2. NER 노트북에서 모델 경로를 HuggingFace 이름 대신 이 Volume 경로로 지정합니다.
3. `HF_HUB_OFFLINE=1` 환경변수를 설정해 온라인 조회를 차단할 수 있습니다.

> S1/S2/S5와 거버넌스·평가는 HuggingFace 다운로드가 필요 없습니다. **NER 단계(S3/S4)만** 이
> 항목의 영향을 받습니다.

## ⑥ quota · 사전 체크리스트

실행 직전 아래를 한 번 더 점검하세요.

**Quota / 용량**

| 자원 | 확인 사항 |
|---|---|
| 컴퓨트 quota | ML 17.3 CPU 클러스터(②의 노드 타입)를 띄울 vCPU quota |
| GPU quota (선택) | ④ qwen 배포 시에만 T4 1장 quota |
| Foundation Model | ③ gpt-oss-120b 호출 가능 여부(region·엔터프라이즈 가용) |

**사전 체크리스트**

- [ ] **카탈로그·스키마·Volume(raw·models) 권한** 확보 또는 관리자 사전 생성 (①)
- [ ] **ML 17.3 LTS 클러스터** 생성 (②, GPU 불필요)
- [ ] **`databricks-gpt-oss-120b` 호출 테스트** 성공 또는 대체 엔드포인트 결정 (③)
- [ ] (선택) **`qwen3-4b-t4` 배포** 후 `qwen_endpoint` 위젯에 입력 (④)
- [ ] (S3/S4) **HuggingFace 다운로드 허용** 또는 모델 Volume 사전 업로드 (⑤)
- [ ] (거버넌스) **`pii_full_access` 계정 그룹** 존재 — 컬럼 마스크 권한 분기 시연용
- [ ] `00_config` 위젯 4개 값 입력 후 **첫 실행** → `FQ=<catalog>.<schema>` 출력 확인
